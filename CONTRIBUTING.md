# Contributing to ROUM Token

Thank you for considering contributing to the ROUM Token project! We welcome contributions from the community that help improve transparency, security, and functionality.

## Code of Conduct

Please read and adhere to our [Code of Conduct](CODE_OF_CONDUCT.md) to maintain a respectful community.

## How to Contribute

### Reporting Bugs

1. **Check existing issues:** Before reporting, search the [issues](https://github.com/Osama-Qonaibe/ROUM-Token/issues) to avoid duplicates
2. **Provide details:** Include:
   - Clear, descriptive title
   - Detailed description of the issue
   - Steps to reproduce
   - Expected vs actual behavior
   - Environment details (OS, Node version, etc.)
   - Screenshots/logs if applicable

### Suggesting Enhancements

1. **Check existing suggestions:** Review [open issues](https://github.com/Osama-Qonaibe/ROUM-Token/issues) for similar requests
2. **Provide rationale:** Explain:
   - Use case and benefits
   - Possible implementation approach
   - Potential impact

### Pull Requests

#### Before You Start

- Fork the repository
- Create a new branch: `git checkout -b feature/your-feature-name`
- Make your changes
- Test thoroughly
- Commit with clear messages: `git commit -m "feat: add new feature"`

#### Commit Message Format

Use [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <subject>

<body>

<footer>
```

Types:
- **feat:** New feature
- **fix:** Bug fix
- **docs:** Documentation changes
- **style:** Code style changes (formatting, semicolons, etc.)
- **refactor:** Code refactoring
- **perf:** Performance improvements
- **test:** Adding or updating tests
- **chore:** Build, dependencies, or tool changes

#### PR Checklist

Before submitting a pull request:

- [ ] Code follows the project's style guidelines
- [ ] Changes are well-documented
- [ ] No breaking changes without discussion
- [ ] Commit messages are clear and descriptive
- [ ] Code has been tested
- [ ] Updated relevant documentation
- [ ] Added tests for new functionality
- [ ] No secrets or sensitive data exposed

### Code Style

#### Solidity

- Follow [Solidity style guide](https://docs.soliditylang.org/en/latest/style-guide.html)
- Use meaningful variable names
- Add comments for complex logic
- Test coverage for all functions

#### Documentation

- Clear, concise markdown
- Code examples where applicable
- Up-to-date information
- Proper formatting and structure

### Testing

For smart contract changes:

1. Write tests for all changes
2. Ensure existing tests still pass
3. Test on BSC testnet before mainnet
4. Verify gas optimization

### Documentation

All contributions should include:

- Updated README if applicable
- Changes to CHANGELOG.md
- Code comments for complex logic

### Review Process

1. Automated checks run on all PRs
2. Code review by maintainers
3. Address feedback and make changes
4. Final approval and merge

## Community Standards

### Be Respectful

- Treat all community members with respect
- Engage in constructive discussion
- Accept constructive criticism
- Acknowledge good ideas from others

### Be Transparent

- Explain your motivations
- Acknowledge limitations
- Be honest about conflicts of interest
- Disclose any personal benefits

### Be Honest

- Don't misrepresent your work
- Acknowledge sources and inspiration
- Be truthful in discussions
- Report issues accurately
